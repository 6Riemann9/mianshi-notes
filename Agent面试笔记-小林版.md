## 1. 什么是 Agent？与大模型有什么本质不同？

**核心要点**
- 普通 LLM 三大局限：知识被冻结（不能获取实时信息）、不能行动（只能生成文字）、没有持续状态（每次调用失忆）
- Agent 核心闭环：**感知 → 规划 → 行动 → 再感知**
- 三件核心支撑：工具调用（Tool Use）、记忆机制、多步推理和自我纠错
- 关键认知：模型只负责决策"调什么工具"，代码负责真正执行——"决策和执行分离"
- A2A 协议（Google 2025.4）：Agent 间通信标准。**MCP 管 Agent↔工具，A2A 管 Agent↔Agent，互补。**

**面试踩雷** ❌ 把 Agent 等同于"插件"或"工具调用"——工具调用只是手段，**自主闭环**才是核心。

---

## 2. Agent 的基本架构由哪些核心组件构成？

**四个核心组件（类比：公司部门）**

| 组件 | 类比 | 职责 |
|------|------|------|
| LLM | 老板 | 所有决策，判断下一步做什么 |
| 工具系统 | 外包团队 | 真正执行搜索、代码、API调用 |
| 记忆系统 | 档案室 | 短期记忆（context window）+ 长期记忆（向量DB） |
| 规划模块 | 项目经理 | 把大目标拆成步骤清单 |

**核心运行循环**
```python
def agent_run(user_goal: str):
    plan = llm.plan(user_goal)
    memory = []
    for step in plan:
        action = llm.decide(step, history=memory, long_term=vector_db.search(step))
        if action.type == "tool_call":
            result = tools.execute(action.tool_name, action.args)
            memory.append({"step": step, "result": result})
        elif action.type == "final_answer":
            return action.content
```

**面试踩雷** ❌ 只说 LLM + 工具，漏掉记忆和规划模块 ❌ "记忆就是上下文"——应该是短期(context window) + 长期(向量DB)两层

## 3. Workflow、Agent、Tools 的概念和区别

**核心区分角度：谁来做「下一步该干什么」的决策**

| 维度 | Tools | Agent | Workflow |
|------|-------|-------|----------|
| 决策能力 | 无（只执行） | 有（LLM 动态决策） | 无（代码写死） |
| 确定性 | 高 | 低 | 高 |
| 灵活性 | 低 | 高 | 低 |
| 调试难度 | 容易 | 困难 | 容易 |

**关键洞察**
- 三者不是三选一，而是粒度不同的**三层结构**，可相互嵌套
- 生产环境最主流：**Agentic Workflow**——Workflow 固定主流程骨架 + 在灵活决策节点嵌入 Agent
- Anthropic 的编排模式：Prompt Chaining → Routing → Parallelization → Orchestrator-Workers → Evaluator-Optimizer
- 停止条件必须设计：最大轮次、token 预算上限、超时机制

## 4. 了解哪些 Agent 设计范式？

**三种范式**

| 范式 | 核心 | 优势 | 劣势 |
|------|------|------|------|
| ReAct | 边想边干，走一步看一步 | 灵活度最高 | 长任务容易跑偏 |
| Plan-and-Execute | 先规划再执行 | 结构清晰，不跑偏 | 灵活度不足 |
| Reflection | 执行后加自我评估 | 输出质量提升 | 增加 token 消耗+延迟 |

**Anthropic 核心原则**："能用 Workflow 解决的问题，就不要用 Agent"
- 先从最简单的 Workflow 开始
- 只有当某个节点确实需要灵活决策时，才升级成 Agent

**面试踩雷** ❌ Reflection 不是独立流程，是叠加在前两者上的"质量增强 buff" ❌ 以为 Agent 是生产首选——纯 Agent 在生产里很少用，行为不确定、难调试

## 5. Agent 推理模式有哪些？ReAct 具体怎么实现？

**CoT → ReAct 演进**
- **CoT（思维链）**：纯文字推理，无法和外部世界交互
- **ReAct**：推理+行动交替，Thought → Action → Observation 循环

**关键认知：不是模型自己在循环！**
- LLM 每次只输出下一步的 Thought + Action
- **你的代码**负责解析 Action、执行工具、把 Observation 填回历史，再调用 LLM

**ReAct 两个实战局限**
1. **循环漂移**：走一步看一步，容易被中间信息吸引偏离原始目标
2. **错误传播**：中间某步结果出错，后面全被带跑，没有回头检查机制

**大小模型搭配**：规划用 GPT-4，执行用 GPT-4o-mini，总成本可降低 70-90%

## 6. ReAct、Plan-and-Execute、Reflection 三种范式核心区别与选型

**关键认知：三者解决不同层次的问题**
- ReAct → 单步灵活性问题
- Plan-and-Execute → 长任务跑偏问题
- Reflection → 输出质量不够好的问题

**token 消耗对比（5 步工具调用，每步 2000 token）**
- ReAct：~30000 token（带完整历史，线性增长）
- Plan-and-Execute：~14500 token
- +Reflection：增加 30-100%

**选型口诀**：任务简单用 ReAct，流程长且复杂用 Plan-and-Execute，输出要求高再加 Reflection。
**混合用法**：规划用 Plan-and-Execute 定全局计划，每步用 ReAct 循环执行，最后做 Reflection 检查。

## 7. 复杂任务怎么拆分？

**为什么拆分**：LLM 的 context window 有上限，任务越大中间状态越多 → "桌面太乱" → 出错率上升

**两种思路**

| | 静态拆分 | 动态拆分 |
|---|---|---|
| 做法 | 提前写死步骤 | LLM 自己规划（Plan-and-Execute） |
| 优势 | 行为可预测 | 灵活 |
| 劣势 | 灵活性低 | 规划质量不稳定 |

**并行优化**：识别步骤依赖关系 → 画成 DAG → 无依赖的并发跑 → 关键路径时间可降 40-60%

**拆分验证三标准**：完备性（覆盖全部需求）、独立性（职责边界清晰）、可验证性（每步有验收标准）

## 8. AI Agent 的记忆机制

**四层记忆**

| 类型 | 载体 | 生命周期 | 访问方式 |
|------|------|---------|---------|
| 感知记忆 | 当次输入 | 单次调用 | 即时 |
| 短期记忆 | context window | 一次任务 | 直接读取 |
| 长期记忆 | 向量/关系DB | 持久 | 语义检索 |
| 实体记忆 | 结构化存储 | 持久 | 精确查询 |

**长期记忆三子类型**：情节记忆（事件经历）→ 语义记忆（抽象规律）→ 程序记忆（操作 SOP）

**三个核心工程问题**
1. **存什么**：只存对下次任务有价值的内容（偏好、结论、知识），不存推理过程、原始日志、闲聊
2. **怎么存**：语义内容用向量 DB，结构化偏好用关系 DB，混合存储是主流
3. **什么时候取**：任务前主动检索注入 system prompt + 执行中按需检索（把"查记忆"封装为 Tool）

**记忆整合**：去重 → 冲突消解 → 抽象提炼（情节→语义）

## 9. 长短期记忆系统具体实现

**短期记忆**：context window 里的 messages 列表。每次调 LLM 传**完整历史**，不只最新一条。

**长期记忆核心技术**：Embedding + 向量数据库
- Embedding：把文字转成数字向量，**语义相近 = 向量距离近**
- 向量数据库：支持相似度检索（不是关键词匹配！）

**记忆粒度**
- 太细 → 碎片化，检索时信息不完整
- 太粗 → 噪音多，LLM 被干扰
- 合理：按"一次完整交互"或"一个独立知识点"存

```python
# 长期记忆存储示意
def save_to_long_term(content: str, metadata: dict):
    embedding = client.embeddings.create(input=content, model="text-embedding-3-small").data[0].embedding
    collection.add(embeddings=[embedding], documents=[content], metadatas=[metadata])
```

**记忆衰减**：相关性分数 = 语义相似度 × 时间衰减因子，越久远权重越低

## 10. 什么是 Multi-Agent？

**两个驱动因素**
1. **context window 硬上限**：单个 Agent 信息量超出窗口就遗忘
2. **专业度问题**：一个 Agent 身兼数职，每件事都不够专注

**三种协作模式**
1. **顺序流水线**：A→B→C，工厂流水线
2. **并行扇出**：调度者分发多个子任务并行执行
3. **辩论/评审**：多个 Agent 各自方案，裁判筛选最优

**核心价值**：每个 Agent 的 context 干净（只装自己那块信息），专业度高，可以并行执行

**组织方式**
- **中心化**（Orchestrator 统一调度）：工程主流，好控制好排查
- **去中心化**（Agent 自行协商）：学术探索，工程少用

## 11. Single-Agent vs Multi-Agent 选型

**需要 Multi-Agent 的三类场景**
1. context 要撑爆了（信息量超窗口）
2. 需要不同专业分工
3. 有子任务可以并行

**Orchestrator 三种变体**
- 静态路由：规则写死，"代码→Coder，搜索→Researcher"
- 动态规划：LLM 动态生成任务计划
- 自适应编排：根据 Worker 结果实时调整后续计划

**渐进式演进策略**：Single-Agent 跑通 → 发现瓶颈 → 把瓶颈环节拆成专门的 Worker Agent。**不要一上来就设计复杂多 Agent 系统。**

**去中心化为什么工程不用**：任务分配没协调、执行顺序没保证、失败没有感知、没人确认"任务完成了"

## 12. Agent 记忆压缩方法

**四种方法三维度**

| 方法 | 解决的问题 | 代价 |
|------|-----------|------|
| 滑动窗口 | "历史太长怎么截" | 关键决策可能被硬截断 |
| 摘要压缩 | 截之前先提炼 | 丢失细节 |
| 重要性过滤 | "内容不等价怎么挑" | 需要打分机制 |
| 结构化抽取 | "对话文本是不是最佳载体" | 开发成本最高 |

**最常用组合**：滑动窗口 + 摘要（丢弃前先做一次摘要提炼）

**层级式摘要**：最近 10 轮原文 → 10-50 轮中期摘要 → 50 轮+长期精炼摘要

**Prompt Caching**：计算层优化，和记忆压缩互补。命中缓存的 token 费用约正常输入的 1/10。

## 13. 为什么选择「手搓」Agent 而不是用框架？

**框架三大痛点**
1. 抽象层太多 → 排查困难（stack trace 四五十层）
2. 版本升级 breaking change → 线上稳定性风险
3. 通用性设计 → 隐性性能开销

**手搓核心价值**：完全掌控——链路透明可观测、精确裁剪无冗余、稳定不受外部升级影响

**折中方案（最务实）**：核心逻辑手写（工具调用循环、对话历史管理、错误处理）+ 周边工具借用框架（tracing、文档解析、向量库客户端）

**核心判断**：框架不是问题，**"不理解就依赖"**才是问题。

## 14. 如何赋予 LLM 规划能力？

**三种演进机制**

| 机制 | 核心 | 代价 |
|------|------|------|
| CoT | 把推理步骤写出来，一条路走到底 | 几乎零成本 |
| ToT | 同时探索多条推理路径，边走边评估边剪枝 | 调用次数 3-5x |
| GoT | 图结构推理，不同路径结论可合并复用 | 学术阶段，工程不成熟 |

**工程最常用：Plan-and-Execute**
- Planner：LLM 制定完整步骤清单
- Executor：逐步执行
- Re-planner：每步后检查，必要时调整后续计划

**选型**：CoT 零成本几乎标配；ToT 效果好但成本高；GoT 了解思想即可

## 15. 讲讲 Agent 的反思机制

**核心闭环**：生成 → 评估 → 改进

**两个关键 Prompt 设计**
1. **评估 prompt**：给出具体检查维度（事实/逻辑/完整性/表达），必须有 **"PASS"出口**
2. **改进 prompt**：同时传入原始任务 + 原始输出 + 评估意见，三者缺一不可

**两种粒度**
- 步骤级反思：每步检查，错误早发现但开销大
- 任务级反思：整体检查，开销小但前期错误已浪费

**Multi-Agent 互评**：独立 Critic Agent 审查 → 比自我反思效果好（同一模型有"自洽"偏见）

**防死循环**：必须硬性设置最大轮次（2-3 轮），不能依赖 LLM 自己判断停止

**进阶机制**
- Reflexion：反思 + 记错题本，跨任务积累经验
- LATS：蒙特卡洛树搜索 + 反思
- 辩论式反思：两个 Agent 互辩暴露深层问题

## 16. 多 Agent 协作与动态切换

**两种通信方式**

| | 消息传递 | 共享状态 |
|---|---|---|
| 核心优势 | 解耦（发方不知道接收方） | 直接（读同一个对象） |
| 代表 | 消息队列 | LangGraph State |
| 适用 | Agent 独立运行 | 流水线型任务 |

**两种路由策略**
- **静态路由**：规则写死。稳定可预测，但覆盖不了边缘情况。
- **动态路由**：LLM 判断。灵活但多一次调用，行为不可预测。

**工程最佳实践**：主流程静态路由保底 + 边缘情况 LLM 动态路由兜底

**Handoff 模式**：Agent 自己做完 → 自己决定交给谁（接力棒模式），不需要中央 Orchestrator。适合 Agent 数量少、职责边界清晰场景。
